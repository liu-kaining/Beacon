# 我做了一个会自动读图表、写中文研报的网站

我一直有一个不太好的习惯：看到好文章就收藏。尤其是那种带着漂亮图表的数据文章，比如全球能源消耗、AI 公司融资排名、人口结构变化、各国债务水平、关键矿产分布、科技公司市值变化等等，每一篇看起来都很值得读，每一张图都像能说明点什么。但问题是，收藏之后，我基本就不看了。打开的时候觉得“这篇不错，之后好好读”，然后它就躺进收藏夹，和另外几百篇“之后好好读”待在一起。

时间久了我发现，自己真正缺的不是信息源，而是一个能帮我把信息先处理一遍的系统。于是我做了 Beacon。

**【配图 1：Beacon 网站首页截图】**

建议放在这里。开头讲“我做了 Beacon”之后，马上给读者一个直观印象，最好是一张首页瀑布流截图，让读者先看到最终产品长什么样。

Beacon 是一个自动化的数据洞察网站。简单说，它会定期去抓取 Visual Capitalist 这类网站上的高质量数据文章，把文章里的图表、表格和正文提取出来，再交给 AI 生成中文分析，最后自动发布成一个静态网页。整个过程不需要我每天手动打开网站，不需要复制粘贴，不需要自己翻译，也不需要登录后台发文章，它会自己跑。

这个项目一开始看起来像是“爬虫 + 翻译 + 静态站点”，但真正做下来之后，我觉得它更准确的定位是一条 AI Native 的自动化内容流水线：RSS 负责发现内容，Python 负责抓取和清洗，OpenAI 负责理解和重写，Cloudflare R2 负责保存图片资产，GitHub Actions 负责定时运行，GitHub Pages 负责最终发布。

## 一开始，我只是想少看一点“原始信息”

最开始想做这个项目，是因为我越来越觉得互联网上最不缺的是信息，真正稀缺的是被整理过、解释过、适合自己阅读的信息。很多英文数据文章质量很高，但它们对中文读者并不算友好。不是说看不懂英文，而是阅读成本太高。

比如一篇讲全球铜矿产量的文章，原文可能有一张很漂亮的信息图，再配几段解释，但如果我真的想理解它，就不只是翻译几句话那么简单。我还要知道铜为什么重要，哪些国家是主要生产国，它和新能源、电网、制造业有什么关系，这些数据变化背后意味着什么。

单纯翻译原文，并不能解决这个问题。我希望看到的是另一种内容形态：它先告诉我这篇文章最重要的结论是什么，再把关键数据点列出来，如果原文有表格，就把表格保留下来，如果有专业词，就顺手解释一下，最后再补一段更适合中文读者理解的背景分析。

这件事很适合 AI 做，因为它不是简单翻译，也不是机械摘要，它需要读懂原文，判断重点，重新组织语言，还要把内容变成一个稳定的结构。

## Beacon 的核心：一条 AI Native 内容流水线

Beacon 的完整流程可以理解成一条内容加工流水线：RSS 发现内容，Python 抓取解析，图片和表格处理，OpenAI 生成中文分析，Cloudflare R2 存储图片，Jinja2 渲染静态网页，最后通过 GitHub Pages 自动发布。每一步的分工都很清楚，程序负责那些确定性的事情，比如抓取、清洗、上传、渲染和发布；AI 负责那些需要理解能力的事情，比如提炼重点、解释数据、组织中文表达和补充背景分析。

这也是我理解的 AI Native：不是给传统项目硬塞一个聊天框，也不是让 AI 包办一切，而是重新设计一条工作流，让 AI 成为其中真正有价值的一环。

## 一篇文章在 Beacon 里会经历什么

在 Beacon 里，一篇文章不是简单地“抓到就发布”，它有自己的生命周期。系统会先从 Visual Capitalist 的 RSS Feed 里发现新文章。RSS 是一个很老派但非常好用的东西，它相当于网站主动告诉外界“我更新了”，对自动化系统来说，这比每天去网页上乱爬要优雅很多。

发现新文章之后，Beacon 会进入解析阶段，提取标题、链接、发布时间、正文内容、表格和图片。这里面表格尤其重要，因为很多数据类文章真正有价值的信息就藏在表格里，如果只是粗暴复制网页正文，表格很容易丢掉，所以 Beacon 会把 HTML 表格转换成 Markdown 表格，后面继续交给 AI 使用。

**【配图 4：文章处理生命周期 / 状态流转图】**

建议放在这里。这里已经讲到“一篇文章不是抓到就发布”，非常适合放 spec 里的状态图，让读者看到文章从 RSS 发现、处理中、AI 成功/失败、图片成功/失败，到最终展示的过程。

图片处理是我做这个项目时感受最深的地方之一，因为它一开始看起来很简单，真正做起来才发现很容易翻车。网页里经常有很多图片：logo、头像、广告图、分享卡片、缩略图、正文图。如果随便拿第一张，最后页面大概率会出现一张毫无意义的图。

所以 Beacon 的取图逻辑不是“拿第一张”，而是一层层判断：先从 RSS 正文里找，再看媒体字段，还不行就打开原文页面重新解析。选图时，它会尽量排除 logo、icon、广告素材和分享卡片，优先选择真正承载内容的信息图。

图片选好之后，系统会下载图片，检查尺寸和完整性，再转换成 WebP 格式，上传到 Cloudflare R2。最终网页里引用的是我自己存储在 R2 上的图片，而不是长期依赖原站外链。这样做有两个好处：一是图片更稳定，不怕原站链接变动；二是访问速度和格式都更可控。

## AI 不是翻译员，更像一个初级分析师

图片和正文都准备好之后，AI 才真正开始工作。Beacon 会把文章内容、表格和上下文交给 OpenAI，让它生成一份中文结构化分析。这里有一个很重要的设计：我没有让 AI 随便写一段“读后感”。

如果只是对模型说“帮我总结一下这篇文章”，结果会很不稳定，有时候它写得很散，有时候格式变来变去，有时候重点也不一定对。这样的结果，人可以勉强读，程序却很难稳定处理。所以 Beacon 要求 AI 输出固定结构，比如核心洞察、关键数据、原文表格整理、深度分析和术语解释。

这样一来，AI 生成的内容不只是给人看的，也是给程序用的，后面的页面模板可以稳定地拿这些字段去渲染，而不是每次都猜模型到底写了什么。这也是我做完之后最大的感受之一：调用大模型不难，难的是让它稳定地嵌进工程系统里。真正可用的 AI 工程，不能只停留在“生成一段话”，它最好能输出结构化结果，能被校验，能被存储，能被后续流程继续消费。

## 自动化系统最怕的不是失败，而是假装成功

做自动化项目的时候，我越来越觉得失败并不可怕，真正可怕的是某一步明明失败了，系统却继续往下跑，最后把半成品展示给用户。所以 Beacon 里设计了两道门。

第一道门是“能不能入库”：如果一篇文章的 AI 分析失败了，或者生成结果只是占位内容，它就不会进入正式数据，因为对这个项目来说，没有 AI 分析，就没有发布价值。

第二道门是“能不能展示”：即使一篇文章已经有了有效 AI 分析，如果没有合格的主图，也不会出现在首页和 RSS 里，因为 Beacon 面向的是数据可视化内容，主图不是装饰，而是内容本身的一部分。

这两道门听起来很小，但它们决定了网站最终的质量。自动化不等于无脑跑脚本，越是自动化，越要提前定义好什么叫成功、什么叫失败、什么时候可以进入下一步、什么时候应该停下来。

在 Beacon 里，一篇文章会经历几个状态：刚被 RSS 发现时是待处理，AI 没生成好就先不发布，图片没处理好也先不展示，只有当文字分析和主图资产都准备完整，它才会真正出现在读者面前。这让整个系统更像一条有质量控制的生产线，而不是一个只会定时运行的脚本。

## GitHub 不只是代码仓库，而是自动化操作系统

这个项目里，GitHub 扮演的角色比我一开始想象中更重要，因为它不只是用来放代码。Beacon 的定时任务由 GitHub Actions 运行，Actions 会拉取仓库，构建 Docker 镜像，执行主程序，生成新的数据和网页，然后把更新后的 `posts.jsonindex.htmlfeed.xml` 再提交回仓库。

GitHub Pages 则负责把生成好的静态页面发布出去。换句话说，我没有单独买服务器，没有搭后台管理系统，也没有维护数据库，GitHub 自己就把“运行、存档、发布”这三件事串起来了。

这对个人项目非常重要，很多项目不是写不出来，而是后面养不起、懒得管、服务器一过期就死掉。Beacon 的思路是尽量轻：能静态化就静态化，能用成熟平台就不用自己造基础设施。代码在 GitHub，自动化在 GitHub Actions，页面在 GitHub Pages，图片在 Cloudflare R2，AI 分析由 OpenAI 完成，几个成熟工具组合在一起，就形成了一个低成本、可持续、自动运行的内容系统。

## 最终呈现：像刷画报，也像读研报

最终生成出来的 Beacon，不是一堆 JSON，也不是命令行输出，而是一个可以直接阅读的网站。我希望它的首页有点像数据画报，一打开就能看到很多高质量的信息图，先快速浏览，看看哪些主题感兴趣；当你点开一张卡片后，会进入文章详情，里面有高清图、核心洞察、关键数据、深度解读、术语解释和原文表格。

如果只是想快速了解，看核心洞察就够了；如果想深入一点，可以继续读后面的分析；如果遇到不熟悉的概念，也能顺手看到解释。前端上也做了一些小优化，比如图片不会一次性全部加载，而是用懒加载，等用户滚动到附近时再加载高清图；打开文章详情时是抽屉式界面，不会完全打断浏览；大图可以继续放大查看；搜索框可以根据标题、摘要和分析内容过滤文章。

这些都不是特别复杂的技术，但它们让 Beacon 更像一个真实可用的产品，而不是一个“跑通了”的 demo。

## 这个项目到底有什么用

对我自己来说，Beacon 首先是一个信息过滤器。我不需要每天主动去找文章，它会自动把值得看的内容抓回来，处理好，放到一个统一的网站里，等我有空的时候打开看中文分析就行。

其次，它也是一个知识沉淀工具。浏览器收藏夹最大的问题是收藏时很激动，回看时很痛苦，里面只有标题和链接，没有结构，也没有上下文。Beacon 生成的是完整页面，有主图、有摘要、有数据、有解释，时间久了，它就不只是一个网站，而是一个围绕全球数据趋势积累起来的小型知识库。

更重要的是，它给了我一个很清晰的 AI 使用方式。很多时候我们聊 AI Native，会聊得很玄，好像非得做一个 Agent，或者做一个聊天机器人，才算用了 AI。但我现在越来越觉得，AI 最适合先嵌进具体流程里：程序负责确定性的调度和执行，AI 负责不确定但需要理解能力的那一层。不是让 AI 替代整个系统，而是让 AI 成为系统里最关键的“理解层”。

---

## 最后

Beacon 对我来说不是一个特别宏大的项目。它没有复杂架构，也没有花哨后台，只是把几个我本来就经常做的动作串起来：找文章、看图表、理解数据、整理成中文、保存下来。区别是，现在这些动作大部分都自动化了。

这也是我最近对 AI 工具最真实的感受：AI 不一定要替你完成一个惊天动地的大任务，它更适合先接管那些你重复做、但又有一点理解门槛的事情。Beacon 就是这样一个小尝试。它用 GitHub 跑自动化，用 Cloudflare R2 存图片，用 OpenAI 做理解和生成，用静态网页做最终呈现。整体很朴素，但对我来说足够实用。

也许以后个人网站、知识库、研究笔记、行业周报，都会越来越像这样：人负责定义方向，AI 负责加工内容，自动化系统负责每天默默运行。而 Beacon，就是我给自己做的一个小灯塔。
